# Reviewed raw-counter-read sites in the built guest kernel (task 110, the
# PARAVIRT-CLOCK.md §3.3 reachability gate, x86 half — see
# scan-counter-opcodes.sh). One `artifact:symbol+0xOFFSET` entry per line — the
# artifact, containing function, and the rdtsc/rdtscp instruction's byte OFFSET
# within it (cross-model r10 P1). Per-SITE, not per-function-count: a NEW read is
# a new site, a removed one goes stale, AND a removed+added pair inside one
# function is caught (the offsets change even though the count would not). The
# `#` comments group the sites by function and record the review class.
#
# ARTIFACT QUALIFICATION (cross-model r4 P2): bzImage is three executable
# artifacts and all three run — the real-mode `setup`, the `decompressor`, and
# the kernel proper (`vmlinux`). All three are scanned; the tag namespaces the
# entry, so the same symbol name in two artifacts (`startup_32` is in two) cannot
# spend the other's budget. As of the baseline below, `setup` and `decompressor`
# contain ZERO counter reads — confirmed both by symbol-attributed disassembly
# (4,846 and 5,988 instructions scanned) and, because the real-mode setup's
# 16-bit stream could in principle decode differently, by a raw byte search of
# their executable images for `0F 31` / `0F 01 F9` (zero occurrences of each).
# They therefore contribute no entries, and the scan's stale-entry check keeps
# it that way: the first counter read to appear in either one fails the build.
#
# A site is
# allowlistable ONLY because the retained RDTSC/RDTSCP trap completes it with
# the same work-derived value the pvclock page carries (§4.1 — on x86 a raw
# read is survivable-by-trap, never a determinism hole). The armed scan fails
# the kernel build on any drift, in either direction.
#
# BASELINE: captured 2026-07-15 from the box build of linux-6.18.35 (the r8
# registration-handshake kernel, bzImage e4591ea8), reviewed against the
# extracted source. Offsets are relative to the containing function, so a
# different compiler version (or a change to a function's earlier code) shifts
# them — such a drift fails the armed scan loudly; re-review the drifted sites
# against that build's capture and commit the delta. For a FULL re-baseline
# (a kernel version bump), add a line starting with `# GATE-UNARMED` — the
# scan then prints the new capture and FAILS the build until the reviewed
# baseline is committed and the marker removed (fail-closed; it can never
# ship a silently-green scan).
#
# Review classes (every entry below is one of these; all trap-backstopped):
#   [alt]    runtime-patched alternatives section (rdtsc/rdtscp variants the
#            kernel may splice in; each executes as a trap if reached)
#   [iommu]  Intel DMAR/IOMMU busy-wait timeouts (IOMMU_WAIT_OP) — no IOMMU
#            is exposed to the guest; unreachable in practice, trapped if not
#   [rng]    random_get_entropy()/jitter-entropy timing — trapped, so timing
#            "entropy" is itself work-derived and deterministic
#   [time]   boot calibration / clocksource / sched_clock / delay paths —
#            unselectable or backstopped once harmony_pvclock is active (the
#            delay decision is recorded in IMPLEMENTATION-task110.md)
#   [entry]  syscall/irqentry/fork kstack-offset randomization — trapped,
#            deterministic offsets
#   [perf]   the perf "msr" software event (tsc pseudo-event)
#   [guest]  other-hypervisor guest support compiled in by the Kata base
#            (kvmclock/vmware/vmclock) — never activates under harmony
#            (leaves/MSRs hidden or deny-gp'd), trapped if reached
#   [harmony] our own deliberate pre-registration anchor-freshening rdtsc
#   [fs]     ext4 mballoc criteria timing (mb_stats path) — trapped
#
